# Voice Hotspot — תוכנית עבודה: ציון עסקה + אגנטיות (hand-off ל-Claude Code)

> מסמך זה מפרק את **הרחבה E (ציון עסקה)** ו**הרחבה F (אגנטיות)** מ-`voice-hotspot-roadmap.md`
> למשימות קונקרטיות לביצוע. עקרון-העל נשמר: **מספרים כנים ללא ריפוד; כל יכולת נמדדת;
> מנוע מקומי תמיד עובד; human-in-the-loop — המערכת מציעה, המשתמש מאשר; שאל בעברית עם אופציות בכל החלטה.**

## כללי עבודה ל-Claude Code
- אתר את הקבצים המדויקים בעצמך (`prosody.js`, ה-UI של הפרומפטר, `api/_llm.js`, מנוע הסיכום) — אל תניח נתיב.
- לכל שינוי UI: השתמש בסקיל `.claude/skills/frontend-design/` והצג תוכנית עיצוב לאישור לפני קוד.
- אל תזין מפתחות/טוקנים בעצמך — המשתמש מזין. עצור והדרך כשנדרש חשבון חיצוני (CRM/Calendar).
- אחרי כל בלוק: אימות (diff / צילום מסך / מספר מדידה), ועדכן `ROADMAP.md` + `research/eval/RESULTS.md`.
- אין "גלאי שקר" / "נבואה" — כל ציון הסתברותי בלבד, עם פירוק לגורמים.
- העדף subagents במקביל (ציון / UI / אגנטיות יכולים לרוץ במקביל) עם מוביל אינטגרציה.

# חלק א' — ציון עסקה / מומנטום סגירה (הרחבה E)

## בלוק E0 (30 דק') · מנוע ציון מבוסס-חוקים  ⟵ נקודת הפתיחה
**למה:** אין דאטה מתויג לאמן מודל עדיין — מתחילים מחוקים משוקללים שקופים, ומשדרגים בהמשך.
- **מקור הגורמים והמשקלים: `knowledge/deal-score-factors.md`** (טקסונומיה מלאה מבוססת מחקר Gong/Clari/Salesmotion/MEDDIC — חיובי/שלילי/דינמיקה, עם משקל, cue לזיהוי, ו-single-call מול cross-call). **הציון = רגש + כל השאר** — הרגש הוא קלט אחד מתוך רבים.
- צור מודול `dealscore.js` שמקבל את ה-state הקיים של האיתותים (פלט A: רגשות/סיגנלים, פרוזודיה, דינמיקת שיחה).
- חשב ציון 0–100: `baseline + Σ(משקל גורם × ביטחון × recency) − Σ(משקל סיכון × ביטחון)`, clamp 0–100.
  - **התחל מהגורמים שמזוהים בשיחה בודדת בלבד** (single-call) — אל תנסה לדמות CRM/cross-call עדיין:
    חיובי: pain מזוהה, metrics מפי הלקוח, EB מעורב, next-step קונקרטי, שאלות מחיר/tier/יישום, security/legal, מתחרה מוקדם, היפוך לשון בעלות ("we/our").
    שלילי: אין/מעורפל next-step, התנגדות לא-מטופלת, "we need to figure out…", התנגדות מחיר/שתיקה אחרי מחיר, מתחרה מאוחר.
  - **דינמיקה (פלח לפי סוג שיחה!):** יחס דיבור (discovery ~46–57% / demo ~65%), מונולוג rep >76s = שלילי, שאלות ~15–16, interactivity ≥5, timing מחיר 38–46 דק', סגמנט next-steps ~4 דק'.
  - **stacking על-אדיטיבי:** 2–3 גורמים חזקים יחד שווים יותר מהסכום. **recency שולט** — איתות טרי גובר על ציון ישן.
- פלט: `{ score, trend: up|flat|down, topFactor: "<הגורם המשפיע ביותר כרגע>" }` — תמיד הצג topFactor, לא רק מספר.
- שמור את כל המשקלים בקובץ קונפיג אחד (`dealscore.config.js`) לכיול קל; הסמן לכל גורם אם הוא single-call או cross-call (cross-call כבוי עד שתהיה שכבת זיכרון/CRM).
- **Acceptance:** `node --check`; בדיקת יחידה עם 3 קלטים סינתטיים (חזק/בינוני/חלש) שמחזירים ציונים הגיוניים + topFactor נכון.

## בלוק E1 (45–60 דק') · מד חי בפרומפטר
- הוסף לפרומפטר מד ציון חי: צבע (ירוק/צהוב/אדום) + חץ מגמה + שורת "הגורם המשפיע ביותר כרגע".
- עדכון עם debounce (השתמש במנגנון ה-debounce הקיים) כדי שלא יקפוץ.
- כבה-בלחיצה (אופציה בהגדרות).
- **Acceptance:** דמו חי + צילום מסך של המד מתעדכן לאורך שיחת בדיקה.

## בלוק E2 (45 דק') · עקומת ציון בסקירה שאחרי השיחה
- במסך הסיכום: גרף קו של הציון לאורך הזמן + סימון רגעי מפנה (איפה עלה/צנח) עם הסיבה.
- **Acceptance:** צילום של עקומה אמיתית מדמו מלא.

## בלוק E3 (מאוחר, דורש דאטה) · כיול ומדידה
- אסוף דאטה מתויג (שיחה → נסגר/לא) — ולו קטן ידני בהתחלה.
- מדוד מתאם בין הציון לתוצאה; כייל ספים/משקלים; שקול מעבר ממודל-חוקים למודל נלמד.
- **Acceptance:** דוח מתאם + טבלת ספים מכוילת ב-`RESULTS.md`. **אל תפרסם מספר לפני שיש מדידה אמיתית.**

# חלק ב' — אגנטיות, שלבים 1–2 (הרחבה F)

> שלב 3 (CRM/Calendar) **לא** בתוכנית הזו — דורש OAuth/חיבור חשבונות; ייפתח בנפרד אחרי אישור.

## בלוק F1 (45 דק') · פעולות תוך-כדי-שיחה (קל, ללא תלות חיצונית)
- כפתורי פעולה מהירים בפרומפטר: "רשום הערה", "סמן התנגדות", "Bookmark לרגע".
- כל פעולה נשמרת מקומית ומקושרת לחותמת-זמן בציר הזמן של השיחה.
- **Acceptance:** הוספת 3 הערות בדמו → מופיעות בסקירה עם הזמנים הנכונים.

## בלוק F2 (60–90 דק') · חבילת אחרי-שיחה (הליבה) — הכל כטיוטה לאישור
- בסיום שיחה הפק אוטומטית:
  1. **סיכום** קצר (כבר קיים מנוע סיכום — חבר אליו).
  2. **טיוטת מייל פולואו-אפ** מותאמת למה שנאמר בשיחה (דרך הראוטר `api/_llm.js`).
  3. **רשימת next-steps** מהשיחה.
  4. **משימת תזכורת** (אובייקט מקומי עם תאריך יעד).
- הכול **ניתן לעריכה** במסך לפני שמירה/העתקה. כפתור "העתק מייל" / "העתק סיכום".
- לוג מקומי של מה הופק (audit), ומתג כיבוי מלא בהגדרות.
- **Acceptance:** דמו מלא → צילום של טיוטת מייל + next-steps + משימה, וכפתור העתקה עובד.

## בלוק F3 (15 דק') · הכנה לשלב 3 (תיעוד בלבד)
- תעד אילו אינטגרציות חיצוניות נרצה (CRM כלשהו / Google Calendar / Tasks) ואילו הרשאות נדרשות.
- **אל תממש** — רק רשימת דרישות + נקודות עצירה לאישור המשתמש. עדכן ב-ROADMAP.

# חלק ג' — שכבת תצוגה חיה (תבניות UI מנצחות)  ⟵ חדש

> **תובנה מהמחקר:** Gong (מוביל השוק) הוא **post-call** — הוא לא קו-פיילוט חי. הקטגוריה החיה היא
> Clari Copilot / Balto / Cresta / Attention. כלומר **הליבה שלנו (זמן אמת) היא בדיוק החלל ש-Gong לא ממלא** —
> וזה מאשש שהכיוון שלנו נכון. הבלוקים כאן מאמצים את תבניות התצוגה החיה שהוכחו אצל הקו-פיילוטים, מותאמות אלינו.

## בלוק G1 (60–90 דק') · כרטיסי תגובה מותָנֵי-טריגר (Battlecards)
- כשמילת-טריגר נאמרת ע"י הלקוח → קופץ כרטיס רלוונטי **מיידית** (התבנית הדומיננטית בשוק; Clari מגיע עם 7 כברירת מחדל).
- 3 קטגוריות: **מחיר** (טריגר: "יקר", "תקציב", "ROI"), **מתחרה** (שם מתחרה → talk-track בידול), **מוצר/התנגדות** (אינטגרציה/use-case).
- מקור הטריגרים: `knowledge/objection-handling.md` + `knowledge/buying-signals.md` (כבר קיימים).
- **Acceptance:** דמו שבו אמירת "זה יקר לי" מפעילה כרטיס מחיר תוך < 300ms.

## בלוק G2 (30 דק') · 3 תגובות לכל התנגדות (הנציג בוחר)
- כל כרטיס התנגדות מציג **3 ניסוחים** שונים לטיפול (תבנית Balto "Dynamic Prompts") — לא שורה אחת קנית. נותן לנציג שליטה.
- **Acceptance:** כרטיס מחיר מציג 3 talk-tracks; קליק בוחר אחד ומסמן אותו.

## בלוק G3 (45 דק') · צ'ק-ליסט חכם עם סימון אוטומטי
- רשימת "מה לכסות בשיחה" (פתיח, שאלות גילוי, כאב, תקציב, next-step) — כשהנציג אומר פריט, ה-AI מזהה ו**מסמן אותו אוטומטית** (חיזוק חיובי מיידי; תבנית Balto/Cresta).
- **Acceptance:** דמו — אמירת שאלת תקציב מסמנת את הפריט בצ'ק-ליסט.

## בלוק G4 (45 דק') · מד דיבור/מונולוג חי
- מד יחס דיבור חי (פלח לפי סוג שיחה: discovery ~46–57% / demo ~65%), התראת "מונולוג ארוך מדי" מעל ~76 שניות, ומדד interactivity (החלפות דובר/דקה, יעד ≥5). מתבסס על מספרי Gong ב-`deal-score-factors.md`.
- **Acceptance:** המד עולה לאדום כשהנציג מדבר רצוף > 76 שניות בדמו.

## בלוק G5 (60 דק', דורש דאטה משתמש) · הצפת תשובות מבסיס ידע (RAG)
- שליפת תשובה מצוטטת ממה שהמשתמש הזין ("מה אני מוכר" + playbook) כשעולה שאלה — כדי שהנציג לא יחפש באמצע שיחה (תבנית Balto AI Answers / Cresta Knowledge Agent).
- **Acceptance:** שאלת לקוח על פיצ'ר → מופיעה תשובה מבסיס הידע של המשתמש.

> **תקציב לטנציה:** הרף בשוק הוא < 200ms (Cresta). יעד אצלנו: כרטיס מופיע בזמן שהנושא עדיין חי. כל בלוק כאן נמדד גם בלטנציה.

# חלק ד' — אינטגרציות שהמערכת צריכה  ⟵ חדש

> זו ההרחבה של שלב 3 באגנטיות (F3). כל אינטגרציה דורשת חיבור חשבון — **עצור והדרך את המשתמש**, אל תזין מפתחות.

1. **STT בלטנציה נמוכה** (Deepgram/אחר) — תנאי-קדם לתצוגה החיה (Web Speech לא מספיק מהיר/מדויק לכל השפות). 🟡 מפתח.
2. **CRM** (אחד להתחלה — HubSpot חינמי הכי קל) — דחיפת הערות/סיכום/next-steps אוטומטית; משיכת הקשר לקוח (סוגר חלק מפער ה-cross-call של ציון העסקה). 🟡 OAuth.
3. **Calendar / Tasks** — יצירת פולואו-אפ ומשימות מה-next-step. 🟡 OAuth.
4. **שליחת מייל** — שליחת טיוטת הפולואו-אפ בפועל (F2 מייצר טיוטה; כאן שולחים). 🟡 OAuth.
5. **לולאה סגורה (closed loop)** — ניתוח אחרי-שיחה מכוון אילו טריגרים/כרטיסים יקפצו בשיחה הבאה (תבנית Balto). 🔵 שלנו.

לכל אינטגרציה: לתעד הרשאות מינימליות, נקודת עצירה לאישור משתמש, ומצב "כבוי" כברירת מחדל.

## סדר ביצוע מוצע
1. E0 → E1 (ציון חי — הכי מבדל, ערך מיידי).
2. G1 → G2 (כרטיסי טריגר + 3 תגובות — תבנית מנצחת, ערך מיידי, ללא תלות חיצונית).
3. F1 → F2 (אגנטיות בלי תלות חיצונית — ערך מיידי).
4. G3 → G4 (צ'ק-ליסט + מד דיבור).
5. E2 (עקומה בסקירה).
6. אינטגרציות (חלק ד') — מתחילים מ-STT, אז CRM אחד; שאר רק אחרי החלטה.
7. E3 + F3 + G5 (כיול, תיעוד שלב 3, RAG) כשיש דאטה/חשבונות מחוברים.
